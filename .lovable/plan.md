

## تشخيص المشكلة — لماذا لم يُطبق REVOKE على الإنتاج

### السبب الجذري

عند النشر، منصة Lovable Cloud تعمل بالترتيب التالي:

```text
1. تطبيق هجرات المطور (بما فيها REVOKE) → timestamp: 183108, 183446
2. تشغيل pg_dump migration (إعادة إنشاء جميع الدوال) → timestamp: 183624
```

**المشكلة**: خطوة pg_dump تُعيد إنشاء الدوال عبر `CREATE OR REPLACE FUNCTION`، وهذا يُعيد الصلاحيات الافتراضية (`EXECUTE` لـ `PUBLIC` الذي يشمل `anon`). فيتم **إلغاء تأثير REVOKE** تلقائياً.

بمعنى آخر: الإصلاح يُطبق ثم يُبطل فوراً.

### الحل

إنشاء هجرة جديدة بـ timestamp أحدث بحيث تُنفذ **بعد** pg_dump في النشرة القادمة. نفس محتوى REVOKE/GRANT لكن بتاريخ جديد.

### التفاصيل التقنية

- إنشاء ملف هجرة جديد يحتوي:
  - `REVOKE EXECUTE ON ALL FUNCTIONS IN SCHEMA public FROM anon;`
  - `GRANT EXECUTE` للدوال الخمس العامة فقط
- عند النشر، الترتيب سيكون:
  1. pg_dump (يعيد الدوال)
  2. الهجرة الجديدة (تسحب الصلاحيات) ← **هذه المرة ستبقى**

### ملاحظة مهمة

قد تتكرر المشكلة في كل نشرة مستقبلية إذا كان pg_dump يعمل دائماً. لذلك سنضيف أيضاً `ALTER DEFAULT PRIVILEGES` لمنع منح صلاحيات تلقائية لـ `anon` على الدوال المستقبلية.
