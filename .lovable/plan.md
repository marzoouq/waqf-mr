## EF-07: خطأ Rate Limit المقلوب في WebAuthn ⚠️ خطأ حقيقي مؤكد

### الدليل المباشر من قاعدة البيانات

دالة `check_rate_limit` تُعيد:
- `true` ← **تم تجاوز الحد** (count > limit)
- `false` ← **ضمن الحد المسموح**

### الاستخدام في الكود

**ai-assistant سطر 48:** `if (isLimited)` → يحظر عند `true` → ✅ **صحيح**

**webauthn سطر 81 (register-options):**
```text
const { data: rlAllowed } = await admin.rpc("check_rate_limit", ...);
if (rlError || rlAllowed === false)  ← يحظر عند false = ضمن الحد!
```
❌ **مقلوب** — يحظر المستخدمين الطبيعيين ويسمح لمن تجاوز الحد

**webauthn سطر 199 (auth-options):**
```text
const { data: allowed } = await admin.rpc("check_rate_limit", ...);
if (rlError || allowed === false)  ← نفس الخطأ
```
❌ **مقلوب** — نفس المشكلة بالضبط

### لماذا لم يُكتشف؟
المتغير سُمّي `rlAllowed` / `allowed` مما أوحى أن الدالة تُعيد "هل مسموح" بينما تُعيد فعلياً "هل تم تجاوز الحد". والتعليق يقول "Fail-closed" لكن المنطق الفعلي fail-open.

### التأثير العملي
- **register-options:** أول طلب تسجيل بصمة يمرّ (لأن count=1 < limit=10 → false → يُحظر!)، لكن بعد تجاوز الحد (count=11 → true → يمرّ!)
- **auth-options:** نفس السلوك — المصادقة بالبصمة تُحظر للمستخدم العادي وتُسمح للمهاجم

### الإصلاح
سطران فقط — تغيير `=== false` إلى `=== true`:
- سطر 81: `if (rlError || rlAllowed === true)`
- سطر 199: `if (rlError || allowed === true)`

**أو** الأوضح: إعادة تسمية المتغيرات لتتوافق مع المعنى الحقيقي:
```text
const { data: isLimited } = await admin.rpc("check_rate_limit", ...);
if (rlError || isLimited)
```

### هل يسبب تعارض؟
**لا** — الإصلاح يغيّر شرط if فقط. لا يؤثر على باقي المنطق.

## EF-01: تسجيل الدخول التجريبي في update_password — تحسين غير عاجل

### الكود الحالي (سطر 163-195)
بعد `updateUserById` الناجح:
1. يُسجّل دخول تجريبي بالبريد وكلمة المرور الجديدة عبر REST API
2. إذا فشل → يُلقي خطأ "فشل تحديث كلمة المرور"
3. إذا نجح → يُسجّل خروج فوري للجلسة التجريبية

### المشكلة
- يُنشئ جلسة مؤقتة في `auth.sessions` بلا داعٍ
- يُسجّل في auth logs دخول/خروج وهمي
- لكنه يخدم غرضاً: التأكد أن Supabase Auth لم يرفض كلمة المرور صامتاً (مثل قائمة HaveIBeenPwned)

### هل يسبب تعارض عند الحذف؟
**خطر منخفض لكنه موجود**: إذا أضاف Supabase فحص pwned passwords في المستقبل بدون إرجاع خطأ من `updateUserById`، سنفقد طبقة التحقق. **التوصية: تأجيل هذا الإصلاح.**
