

# فحص جنائي شامل لنظام الامتثال لهيئة الزكاة والضريبة والجمارك (ZATCA)

---

## نتائج الفحص الجنائي

### ✅ البنية المعمارية — ممتازة (95%)

النظام يتكون من 4 طبقات متكاملة:

```text
┌──────────────────────────────────────────────────┐
│ UI Layer                                          │
│  InvoicesPage → إدارة الفواتير العامة             │
│  ZatcaManagementPage → إدارة ZATCA الشاملة       │
│  ZatcaSettingsTab → إعدادات التسجيل الضريبي      │
│  InvoicePreviewDialog → معاينة الفاتورة          │
│  InvoiceStepsGuide → دليل مرئي من 6 خطوات       │
├──────────────────────────────────────────────────┤
│ Hooks Layer                                       │
│  useInvoices → CRUD + Magic Bytes + Signed URLs   │
│  usePaymentInvoices → فواتير الدفعات التلقائية    │
├──────────────────────────────────────────────────┤
│ Edge Functions (Backend)                          │
│  zatca-xml-generator → UBL 2.1 XML               │
│  zatca-signer → ECDSA P-256 + XMLDSig + QR TLV   │
│  zatca-api → CSR/Onboard/Compliance/Report/Clear  │
│  generate-invoice-pdf → PDF A4 عربي + QR          │
├──────────────────────────────────────────────────┤
│ Database Layer                                    │
│  invoices, payment_invoices, invoice_chain         │
│  zatca_certificates (RLS admin-only)              │
└──────────────────────────────────────────────────┘
```

### ✅ دورة حياة الفاتورة الإلكترونية — مكتملة

1. **إنشاء العقد** → توليد فواتير تلقائية عبر `generate_contract_invoices` RPC
2. **توليد XML** → `zatca-xml-generator` ينشئ UBL 2.1 كامل
3. **التوقيع** → `zatca-signer` يوقع ECDSA P-256 مع XMLDSig + XAdES
4. **الإبلاغ/الاعتماد** → `zatca-api` يرسل للهيئة (report/clearance)
5. **تحميل PDF** → `generate-invoice-pdf` ينشئ PDF مع QR + خطوط عربية

### ✅ نقاط القوة المؤكدة

| المكون | النتيجة | التفاصيل |
|--------|---------|----------|
| XML Generator | 97% | UBL 2.1 كامل، Standard (0100000) + Simplified (0200000)، Credit/Debit Notes |
| Digital Signer | 96% | C14N + ECDSA P-256 + XAdES SignedProperties + X509 IssuerSerial parsing |
| QR TLV | 98% | Tags 1-9 مع BER Length encoding للقيم > 127 بايت |
| Invoice Chain | 99% | ICV ذري + PIH (Previous Invoice Hash) = سلامة السلسلة |
| CSR Onboarding | 95% | ASN.1 DER يدوي + SAN + CertificateTemplateName |
| File Security | 99% | Magic Bytes + Path Traversal + Signed URLs (300s TTL) |
| Auth | 100% | getUser() + role check في كل Edge Function |

## ⚠️ الثغرات والتحسينات المطلوبة

### 1. قالب الفاتورة الضريبية — تطوير مطلوب (الأولوية العليا)

**الحالي**: `InvoicePreviewDialog` يعرض فاتورة مبسطة بدون:
- شعار المنشأة (مكان "ضع شعارك هنا" في الصورة المرجعية)
- بيانات المشتري التفصيلية (العنوان، الرقم الضريبي) للفواتير القياسية
- رمز QR الفعلي (TLV Base64)
- تمييز بصري بين "فاتورة ضريبية" و "فاتورة ضريبية مبسطة"
- رقم التوقيع/التسلسل ومعلومات ZATCA UUID

**المطلوب** (مستوحى من لوحة "الأستاذ" المرفقة):
- قالب احترافي مع ترويسة تحتوي على شعار + بيانات البائع
- قسم مخصص لبيانات المشتري (الاسم، العنوان، الرقم الضريبي، السجل التجاري)
- جدول بنود تفصيلي (البند، الوصف، الكمية، السعر، المجموع بدون ضريبة، نسبة الضريبة، المجموع)
- قسم الإجماليات (قبل الضريبة، القيمة المضافة، الإجمالي، المستحق)
- رمز QR فعلي من بيانات ZATCA
- تذييل مع توقيع رقمي وملاحظات

### 2. AllowanceCharge مفقود في XML

**الحالي**: لا يوجد دعم لـ `AllowanceCharge` في UBL XML
**التأثير**: لا يمكن إصدار فواتير مع خصومات أو رسوم إضافية
**التوصية**: إضافة عنصر AllowanceCharge اختياري في `buildUBL()`

### 3. بنود متعددة (Multi-line Items)

**الحالي**: `InvoiceLine` واحد فقط (hardcoded `<cbc:ID>1</cbc:ID>`)
**التأثير**: مناسب لفواتير الإيجار، لكن فواتير الصيانة/المرافق قد تحتاج بنوداً متعددة
**التوصية**: تعديل `buildUBL()` لدعم مصفوفة بنود

### 4. تمييز بصري للحقول الإلزامية الناقصة

**الحالي**: يوجد بالفعل في `ZatcaManagementPage` (شرط `missingSettings`)
**التحسين**: إضافة مؤشر أحمر في `InvoicePreviewDialog` عند نقص بيانات المشتري للفواتير القياسية

## خطة التنفيذ

### المرحلة 1: تطوير قالب الفاتورة الاحترافي

**الملف**: `src/components/invoices/InvoicePreviewDialog.tsx`

1. **تحسين الترويسة**: إضافة مكان الشعار + تمييز نوع الفاتورة (ضريبية/مبسطة) بلون مختلف
2. **بيانات المشتري**: إضافة قسم كامل لبيانات المشتري (العنوان، الضريبي، السجل التجاري) يظهر فقط للفواتير القياسية
3. **جدول البنود المطور**: إضافة أعمدة إضافية (المجموع بدون الضريبة، نسبة الضريبة) مع تنسيق احترافي
4. **رمز QR حقيقي**: توليد QR من بيانات TLV Base64 باستخدام مكتبة `qrcode` وعرضه في الفاتورة
5. **بيانات ZATCA**: عرض UUID + ICV + حالة التوقيع عند توفرها
6. **مؤشرات النقص**: تلوين الحقول الفارغة الإلزامية بالأحمر للفواتير القياسية

### المرحلة 2: تحسين بنية البيانات في InvoicesPage

**الملف**: `src/pages/dashboard/InvoicesPage.tsx`

1. تحديث `buildPreviewData()` لإضافة بيانات المشتري من العقد (العنوان، الضريبي)
2. إضافة حقول ZATCA (UUID، ICV، zatca_status) إلى بيانات المعاينة
3. إضافة QR TLV data من `zatca_xml` إذا كان متوفراً

### المرحلة 3: نظام قوالب الفواتير

**ملف جديد**: `src/components/invoices/InvoiceTemplates.tsx`

1. قالب "الضريبي الاحترافي" — مطابق للصورة المرفقة (ترويسة خضراء، جدول بنود، QR)
2. قالب "المبسط" — للفواتير B2C بدون بيانات مشتري تفصيلية
3. إمكانية التبديل بين القوالب من واجهة المعاينة

### المرحلة 4: تكامل QR في المعاينة

**تثبيت**: إضافة مكتبة `qrcode.react` لعرض QR مباشرة في المعاينة
1. استخراج TLV Base64 من `zatca_xml` المُوقّع
2. عرض QR كمكون React SVG في أسفل الفاتورة
3. Fallback لتوليد QR أساسي (Tags 1-5) عند عدم وجود XML موقّع
