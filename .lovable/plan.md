

# نظام قوالب الفاتورة المتعددة — خطة التنفيذ

## المشكلة الحالية
الفاتورة الحالية عبارة عن جدول بسيط (بيان/تفاصيل) بدون تفاصيل البائع أو بيانات الدفع أو جدول بنود احترافي. المطلوب: قوالب متعددة بنفس البيانات الأساسية لكن بتصميمات مختلفة.

## التصميم المقترح

### القوالب الثلاثة

| القالب | الوصف | الاستخدام |
|--------|-------|-----------|
| **كلاسيكي** (الحالي مُحسَّن) | جدول key-value + بيانات البائع/المشتري + بنك | فواتير بسيطة بدون ضريبة |
| **ضريبي احترافي** | مثل المرجع المرفق: ترويسة بائع + QR + جدول بنود (8 أعمدة) + ملخص ضريبي + بيانات دفع | فواتير ZATCA الضريبية |
| **مختصر** | فاتورة صفحة واحدة مضغوطة بدون إطار زخرفي | طباعة سريعة |

### البيانات الأساسية المشتركة (لا تتغير بين القوالب)
- بيانات البائع (اسم الوقف، الرقم الضريبي، السجل التجاري، العنوان)
- بيانات الفاتورة (رقم، تاريخ، عقد، عقار، مستأجر)
- المبالغ (قبل الضريبة، الضريبة، الإجمالي)
- QR Code (للفواتير الضريبية)
- بيانات الدفع (البنك، الحساب، IBAN)

## التغييرات المطلوبة

### 1. توسيع `PdfWaqfInfo` — `src/utils/pdf/core.ts`
إضافة حقول جديدة:
```typescript
commercialReg?: string;    // السجل التجاري
address?: string;          // العنوان الكامل
bankName?: string;         // اسم البنك
bankAccount?: string;      // رقم الحساب
bankIBAN?: string;         // رقم الآيبان
```

### 2. تحديث `usePdfWaqfInfo` — `src/hooks/usePdfWaqfInfo.ts`
قراءة الحقول الجديدة من `app_settings` (موجودة فعلاً: `commercial_registration_number`, `business_address_*`)

### 3. إضافة `PaymentInvoicePdfData.tenantVatNumber` و `tenantAddress`
حقول اختيارية لبيانات العميل/المشتري في الفاتورة الضريبية

### 4. إعادة هيكلة `paymentInvoice.ts`
- إضافة `InvoiceTemplate = 'classic' | 'tax_professional' | 'compact'`
- دالة `generatePaymentInvoicePDF` تقبل `template` parameter
- ثلاث دوال داخلية: `renderClassic()`, `renderTaxProfessional()`, `renderCompact()`
- الدوال تتشارك: `renderSellerInfo()`, `renderBuyerInfo()`, `renderLineItems()`, `renderVatSummary()`, `renderBankDetails()`, `renderQR()`
- منطق الرفع والتخزين يبقى مشتركاً
