
# تشخيص جنائي لخطأ SQL Editor

## النتيجة المؤكدة الآن

الخطأ ليس بسبب فشل الإصلاح، بل لأنك حاولت تشغيل `UPDATE` على سجلات مرتبطة بسنة مالية **مقفلة**، ويوجد Trigger حماية يمنع أي تعديل على بيانات السنوات المقفلة:

```sql
prevent_closed_fiscal_year_modification()
```

ورسالة الخطأ جاءت من هذا الحاجز الأمني نفسه:

```text
لا يمكن تعديل بيانات سنة مالية مقفلة
```

---

## ما الذي تحققته الآن

### 1) بيانات التوزيعات في الإنتاج
تم التحقق مباشرة من قاعدة البيانات:

- عدد التوزيعات الكلي: **14**
- عدد السجلات التي ما زال فيها `fiscal_year_id IS NULL`: **0**

هذا يعني أن **تصحيح البيانات تم بالفعل**، ولم تعد هناك توزيعات ناقصة السنة المالية.

### 2) السنة المنشورة
الموجود حالياً:

- `2024-2025` → `closed` و `published = true`
- `2025-2026` → `active` و `published = false`

### 3) إصلاح الأمان
سياسة `SELECT` على `distributions` تحتوي الآن على:
- beneficiary
- admin
- waqif
- accountant

إذن **تحذير المحاسب تم إصلاحه** أيضاً.

### 4) دالة التعيين التلقائي
الدالة `set_distribution_fiscal_year()` موجودة في الإنتاج وبها:
```sql
SET search_path TO 'public'
```

---

## الاستنتاج

### السبب الحقيقي للخطأ
أنت شغّلت أمر تعديل بيانات مثل:

```sql
UPDATE distributions ...
```

على صفوف تعود لسنة `2024-2025` وهي سنة **مقفلة**، فتم رفض العملية من Trigger الحماية.

### هل ما زال الإصلاح مطلوباً؟
**لا، ليس بنفس أمر الـ UPDATE السابق**، لأن:
- البيانات المصححة موجودة فعلاً (`NULL = 0`)
- إعادة تشغيل أمر التعديل على سنة مقفلة ستفشل دائماً

---

## الخطة الصحيحة الآن

### A. أوقف إعادة تشغيل أمر `UPDATE distributions`
لا تعِد تنفيذ أمر تصحيح البيانات مرة أخرى في SQL Editor، لأنه:
- لم يعد مطلوباً
- وسيصطدم بقفل السنة المالية

### B. استخدم فقط أوامر تحقق قراءة `SELECT`
للتأكد من الوضع الحالي في الإنتاج، استخدم أوامر قراءة فقط مثل:

```sql
SELECT count(*) AS total_count,
       count(*) FILTER (WHERE fiscal_year_id IS NULL) AS null_fy_count
FROM public.distributions;
```

```sql
SELECT id, label, status, published
FROM public.fiscal_years
ORDER BY created_at DESC;
```

```sql
SELECT id, beneficiary_id, account_id, fiscal_year_id, amount, date
FROM public.distributions
ORDER BY created_at DESC
LIMIT 20;
```

### C. إذا كان الهدف نقل “باقي” الإصلاحات من التطوير إلى الإنتاج
فقسّمها إلى نوعين:

#### 1) إصلاحات بيانات
هذه **لا تُعاد** الآن لأن البيانات أصبحت مصححة بالفعل.

#### 2) إصلاحات بنيوية/دوال/Triggers
هذه تُنقل فقط إذا كان هناك عنصر ناقص فعلاً في الإنتاج، مثل:
- Trigger غير موجود
- Function غير موجودة
- Policy غير موجودة

لكن قبل أي تنفيذ، يجب التحقق أولاً من النواقص الفعلية، وليس إعادة كل SQL بشكل أعمى.

---

## هل الخطة الحالية كاملة؟

نعم، بعد الفحص الحالي الخطة الصحيحة أصبحت:

1. **عدم إعادة تنفيذ UPDATE على التوزيعات**
2. **الاعتماد على SELECT للتحقق فقط**
3. **اعتبار إصلاح `fiscal_year_id` منجزاً بالفعل**
4. **الانتقال إلى التحقق الوظيفي من ظهور البطاقات والتقرير**
5. **فقط إذا ظهر نقص بنيوي فعلي** نضيف SQL منفصل له دون المساس ببيانات السنة المقفلة

---

## ما الذي يغطي ظهور البطاقات مباشرة؟

بما أن:
- `fiscal_year_id` لم يعد `NULL`
- السنة `2024-2025` منشورة
- سياسة القراءة للمحاسب مصححة

فمن ناحية قاعدة البيانات، الخطة **تغطي سبب اختفاء بطاقات المبالغ المستلمة**.

إذا بقيت البطاقات لا تظهر بعد ذلك، فالمشكلة لن تكون في هذا الـ SQL، بل ستكون في أحد المسارات التالية:
- اختيار السنة في الواجهة
- ربط المستخدم بالمستفيد
- فلترة الصفحة أو RPC
- تصدير التقرير نفسه

---

## التفاصيل التقنية

```text
الوضع السابق:
distributions.fiscal_year_id = NULL
→ RLS تمنع القراءة
→ بطاقات المستفيد فارغة

الوضع الحالي:
distributions.fiscal_year_id != NULL
→ السنة المنشورة accessible
→ القراءة مسموحة
→ البطاقات يفترض أن تظهر

لكن:
أي UPDATE جديد على بيانات سنة closed
→ trigger prevent_closed_fiscal_year_modification
→ ERROR P0001
```

---

## القرار التنفيذي الصحيح الآن

بدلاً من تنفيذ SQL تعديل جديد في SQL Editor:

- اعتبر **تصحيح البيانات منجزاً**
- لا تنفذ `UPDATE distributions` مرة أخرى
- انتقل إلى **فحص الإنتاج وظيفياً** للتأكد أن:
  - بطاقة المبالغ المستلمة تظهر
  - صفحة حصتي تعرض القيم
  - تصدير التقرير يقرأ نفس البيانات
