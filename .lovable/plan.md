

# إصلاح مشكلة دخول المستفيدين برقم الهوية - نتائج التحقيق الجنائي

## الأدلة الجنائية المكتشفة

### 1. بيانات قاعدة البيانات (سليمة)
تم التحقق من بيئتي الاختبار والإنتاج: جميع المستفيدين الـ 14 لديهم:
- `national_id` مسجل (مثال: 1014548257)
- `email` مسجل (مثال: 1014548257@waqf.app)
- `user_id` مرتبط بحساب Auth

### 2. وظيفة lookup-national-id (تعمل لكن بطيئة جداً)
- الاختبار المباشر: ارجعت `{"email":"10***@waqf.app","found":true}` -- **نجح**
- زمن الاستجابة: **6,569 مللي ثانية (6.5 ثانية)** -- بطء خطير بسبب Cold Start
- على شبكة جوال بطيئة، هذا قد يتسبب في انقطاع الاتصال قبل وصول الرد

### 3. سجلات الإنتاج (فارغة)
- **صفر** سجلات لوظيفة `lookup-national-id` في بيئة الإنتاج
- هذا يعني: إما لم يتم استدعاؤها مؤخراً، أو أن الاستدعاء يفشل قبل الوصول للوظيفة (مشكلة شبكة/CORS)

### 4. الأرقام العربية-الهندية (ثغرة مؤكدة)
- حقل الإدخال في صفحة الدخول يقبل أي نص
- لوحة المفاتيح العربية على الجوال قد تُدخل: `١٠١٤٥٤٨٢٥٧` بدلاً من `1014548257`
- الـ regex في الوظيفة `(/^\d{10}$/)` يرفض الأرقام العربية-الهندية
- النتيجة: الوظيفة ترجع خطأ `"رقم الهوية يجب أن يكون 10 أرقام"` لكن الكود في Auth.tsx يُظهر `"رقم الهوية غير مسجل في النظام"` لأنه يتعامل مع أي خطأ بنفس الرسالة

## خطة الإصلاح

### الإصلاح 1: تحويل الأرقام العربية-الهندية تلقائياً (Frontend)

**ملف: `src/pages/Auth.tsx`**

إضافة دالة تحويل الأرقام العربية إلى لاتينية قبل إرسال الطلب:

```typescript
function normalizeArabicDigits(str: string): string {
  return str
    .replace(/[٠-٩]/g, d => String.fromCharCode(d.charCodeAt(0) - 0x0660 + 48))
    .replace(/[۰-۹]/g, d => String.fromCharCode(d.charCodeAt(0) - 0x06F0 + 48))
    .trim();
}
```

واستخدامها في `handleSignIn` قبل إرسال رقم الهوية:
```typescript
const cleanId = normalizeArabicDigits(nationalId);
```

### الإصلاح 2: تحويل الأرقام أيضاً في الوظيفة الخلفية (Defense in Depth)

**ملف: `supabase/functions/lookup-national-id/index.ts`**

إضافة نفس دالة التحويل في الوظيفة كطبقة حماية إضافية قبل التحقق بالـ regex.

### الإصلاح 3: تحسين رسائل الخطأ (Frontend)

**ملف: `src/pages/Auth.tsx`**

بدلاً من رسالة واحدة لكل حالات الفشل، التفريق بين:
- خطأ في الشبكة/اتصال -> "حدث خطأ في الاتصال، يرجى المحاولة مرة أخرى"
- رقم غير مسجل (found=false) -> "رقم الهوية غير مسجل في النظام"
- خطأ تنسيق -> "يرجى إدخال رقم الهوية بالأرقام الإنجليزية (10 أرقام)"

تعديل سطر 120:
```typescript
if (lookupError) {
  toast.error('حدث خطأ في الاتصال، يرجى المحاولة مرة أخرى');
  return;
}
if (!data?.found || !data?.email) {
  toast.error('رقم الهوية غير مسجل في النظام');
  return;
}
```

### الإصلاح 4: التحقق من صحة الإدخال قبل الإرسال (Frontend)

إضافة تحقق فوري في حقل الإدخال:
```typescript
if (!/^\d{10}$/.test(cleanId)) {
  toast.error('رقم الهوية يجب أن يكون 10 أرقام إنجليزية');
  return;
}
```

## التأثير المتوقع
- المستفيدون الذين يستخدمون لوحة مفاتيح عربية سيتمكنون من الدخول مباشرة
- رسائل خطأ أوضح تساعد في تشخيص المشكلة الحقيقية
- حماية مزدوجة (frontend + backend) ضد مشكلة الأرقام
